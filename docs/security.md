# Модель угроз и безопасности

Документ описывает критичные активы, актёров, риски и митигации Telegram Lead Aggregator. Основа — ТЗ разд. 4.1, 8, 9 + ADR-0003, ADR-0004.

## 1. Критичные активы

| Актив | Почему критичен | Где хранится |
|-------|-----------------|--------------|
| **Telethon session-файлы** | Компрометация = полный доступ к Telegram-аккаунту, возможность читать/писать от имени пользователя. Восстановление требует нового номера. | На VPS в `/opt/tla/data/sessions/*.enc` (шифрованный AES-256-GCM); офлайн-бэкап у PM. |
| **Anthropic Claude API key** | Компрометация = неограниченные расходы на счёт проекта. | GitHub Environments (`prod`), VPS `/opt/tla/infra/.env.prod` (chmod 600, owner `deploy`). |
| **OpenAI API key** | То же. | Там же. |
| **Telegram API_ID / API_HASH** | Без них Telethon не авторизуется; компрометация позволяет имитировать наше приложение. | Там же. |
| **Aiogram bot token** | Компрометация = злоумышленник может слать сообщения от имени бота Максиму. | Там же. |
| **Postgres credentials + данные лидов** | Данные лидов (имена, username'ы, bio, извлечённые контакты) — PII по GDPR. | Postgres на VPS; dump'ы в `/opt/tla/backups/pg/` и S3 (SSE-S3). |
| **JWT signing key** | Компрометация = подделка сессий админов. | Env на VPS и Vercel. |
| **Vercel deployment token** | Компрометация = деплой произвольного frontend. | GitHub Environments. |
| **SSH-ключи deploy на VPS** | Компрометация = rootkit на prod. | Локальные ключи разработчиков, authorized_keys на VPS. |

## 2. Акторы угроз

| Актор | Мотивация | Вероятная вектор атаки |
|-------|-----------|------------------------|
| **Telegram Anti-Spam** | Выявление автоматизации | Поведенческий анализ, жалобы пользователей → ban аккаунта |
| **Злоумышленник-оппортунист** | Кража API-ключей (майнинг LLM-трафика, спам) | Утечка через repo, exposed `.env`, phishing сотрудника |
| **Конкурент** | Саботаж, кража лидов | Атака на админ-UI (credential stuffing), попытка SQL-инъекции в поиск |
| **Субъект данных (лид)** | GDPR right-to-be-forgotten | Запрос на удаление через email |
| **Инсайдер** | Умышленно или случайно | Неосторожный commit секрета, копирование БД на личное устройство |
| **Провайдер (Hetzner, Vercel, Anthropic)** | Не злоумышленник, но subpoena / breach | Компрометация на уровне провайдера |

## 3. Риски и митигации

### R-01. Бан Telegram-аккаунта

- **Вероятность:** средняя (ТЗ разд. 8). **Влияние:** высокое.
- **Митигация:** ADR-0003; имитация поведения (ТЗ FEATURE-10); ротация 2–3 аккаунтов; мониторинг rate-limit; runbook `telethon-ban.md`.

### R-02. Утечка Telethon session

- **Вероятность:** низкая. **Влияние:** критическое.
- **Митигация:** шифрование на диске (AES-256-GCM, ключ в env VPS); файлы не коммитятся (glob `*.session*` в `.gitignore`); офлайн-бэкап у PM; процедура revoke всех сессий через @SpamBot при подозрении.

### R-03. Утечка Claude / OpenAI API key

- **Вероятность:** низкая (при соблюдении процесса). **Влияние:** финансовое (до $10K за ночь).
- **Митигация:** хранение только в GitHub Environments с ограничением на ветки; `gitleaks` в pre-commit и CI; IP-allowlist на Anthropic console (VPS IP); дневной billing-cap в консоли провайдеров; алерт при аномальном cost-spike (runbook `llm-cost-spike.md`).

### R-04. SQL-инъекция / XSS в админке

- **Вероятность:** низкая (ORM + React). **Влияние:** среднее.
- **Митигация:** SQLAlchemy-параметризация (запрет raw SQL вне миграций); React escapes by default; CSP headers на Next.js; `bandit` и `eslint-plugin-security` в CI.

### R-05. Brute-force / credential stuffing админки

- **Вероятность:** средняя. **Влияние:** высокое.
- **Митигация:** 2FA обязательно для админов (ТЗ FEATURE-09); rate-limit на `/api/v1/auth/login` (5/min/IP); длинные JWT с коротким TTL + refresh; аудит логинов в `audit_log`.

### R-06. Потеря данных (ошибка миграции, disk failure)

- **Вероятность:** низкая. **Влияние:** высокое.
- **Митигация:** ежедневные бэкапы Postgres с retention 30 дней (ТЗ разд. 4.1); офсайт-копия в S3; runbook `db-restore.md`; тест восстановления раз в квартал.

### R-07. GDPR-запрос на удаление

- **Вероятность:** низкая (публичные сообщения, но всё же). **Влияние:** среднее (штрафы + репутация).
- **Митигация:** процедура soft-delete с `deleted_at` и скрипт `backend/scripts/gdpr_purge.py --user-id` для `sender_profiles` и связанных `lead_analysis`; `raw_messages` анонимизируется (обнуление `message_text`, `sender_name`); ответ заявителю в 30 дней.

### R-08. Инсайдерская утечка / случайный commit секрета

- **Вероятность:** средняя (короткий срок, новая команда). **Влияние:** среднее.
- **Митигация:** pre-commit hook с `gitleaks`; CI secret-scan; code review обязателен; секреты никогда не логируются (фильтры в структурном логгере); least-privilege на VPS (deploy user не root).

### R-09. Компрометация Vercel / GitHub Actions

- **Вероятность:** низкая. **Влияние:** высокое.
- **Митигация:** минимальные разрешения токенов; separate environments dev/prod; required reviewers на prod-deploy; отзыв токенов при уходе человека из команды.

### R-10. Репутационный риск от качества лидов

- **Вероятность:** средняя. **Влияние:** высокое.
- **Митигация:** ручная валидация LLM раз в 2 недели (ТЗ FEATURE-05); в этом этапе только уведомления, аутрич — вручную Максимом (ТЗ разд. 8); версионирование промптов с rollback.

## 4. Операционные контроли

- **Audit log** (`audit_log` table): логины админов, изменения `telegram_sources`, `keyword_triggers`, `prompt_versions`, экспорт лидов, GDPR-операции. Хранить минимум 1 год.
- **HTTPS обязательно**: Let's Encrypt + Nginx, HSTS на frontend.
- **Логи не содержат PII**: тексты сообщений — только в `raw_messages`; в stdout логах — только ID и метаданные. Проверяется `tests/test_no_pii_in_logs.py`.
- **Зависимости**: `pip-audit` и `npm audit` в CI, блок high/critical.
- **Доступ к VPS**: только через SSH-ключи, PasswordAuthentication off, root login off, fail2ban.
- **Ротация секретов**: раз в 6 месяцев плановая; немедленно при компрометации или уходе человека.

## 5. Процедура при подозрении на компрометацию

1. Создать инцидент в `#incidents` (severity SEV-1).
2. Revoke скомпрометированного секрета у провайдера (Anthropic/OpenAI/Telegram).
3. Проверить audit log и billing за последние 30 дней.
4. Ротация всех смежных секретов.
5. Постмортем, обновление этого документа и runbook'ов.
