# ADR-0001: Фиксация архитектурных решений через ADR

- **Статус:** Accepted
- **Дата:** 2026-04-24
- **Авторы:** команда Telegram Lead Aggregator

## Контекст

Проект Telegram Lead Aggregator стартует с распределённой командой (backend, frontend, DevOps, PM) и коротким горизонтом планирования: 3 спринта по 2 недели (ТЗ разд. 5). Значимые архитектурные решения (user-session vs Bot API, выбор LLM, место хостинга frontend/backend) принимаются сейчас и будут определять цену ошибки на месяцы вперёд.

Без явной фиксации эти решения растворяются в чатах и PR-комментариях. Новый участник команды не может быстро понять, почему выбран именно такой путь, и начинает переоткрывать уже закрытые вопросы.

## Решение

Мы используем Architecture Decision Records (ADR) в формате Michael Nygard для значимых архитектурных решений. ADR хранятся в `docs/adr/NNNN-slug.md`, нумеруются последовательно, пишутся в момент принятия решения и не редактируются ретроспективно (кроме поля Status — можно перевести в Deprecated/Superseded).

Шаблон — `docs/adr/TEMPLATE.md`. Объём одного ADR — 100–300 слов без счёта служебных полей.

## Рассмотренные альтернативы

- **Wiki/Notion без версионирования.** Отвергнуто: теряется связь решения с кодом и PR, нет ревью через pull request.
- **Решения только в PR-описаниях.** Отвергнуто: PR-описания живут в GitHub отдельно от репо и не индексируются как единый список.

## Последствия

- **Позитивные.** Новые участники получают контекст за минуты. Ревью решений происходит через обычный PR-flow.
- **Негативные.** Накладные расходы на написание ADR (~30 мин на решение).
- **Нейтральные.** Definition of Done требует ADR для значимых решений (см. `docs/dod.md`).
