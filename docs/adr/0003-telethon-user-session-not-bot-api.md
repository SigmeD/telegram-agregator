# ADR-0003: Telethon user-session вместо Bot API

- **Статус:** Accepted
- **Дата:** 2026-04-24
- **Авторы:** команда Telegram Lead Aggregator

## Контекст

Бизнес-задача — мониторить 50+ открытых Telegram-чатов и каналов стартап-сообществ СНГ (ТЗ разд. 1.1, 2.3). Бот Telegram (Bot API) не может читать сообщения в чатах/каналах, куда его не добавили администраторы этих чатов. Большинство целевых сообществ (Pioneers Club, Russian Startups, Startup Jedi и т.д., ТЗ FEATURE-02) не добавят стороннего бота. Без доступа к сообщениям продукт невозможен.

Альтернатива — user-session через MTProto (Telethon): логин обычным пользовательским аккаунтом с отдельным номером, далее чтение всех чатов, в которые этот пользователь вступил.

## Решение

Мы используем Telethon 1.34+ с user-session от выделенного Telegram-аккаунта. Bot API используется только для исходящего канала уведомлений Максиму (Aiogram bot — это ок, т.к. отправитель, а не слушатель).

Рабочий режим — 2–3 аккаунта в ротации, имитация человеческого поведения (задержки 1–5 сек, < 200 API-вызовов/мин), автопауза на rate-limit (ТЗ FEATURE-10).

## Рассмотренные альтернативы

- **Bot API.** Покрывает < 5% целевых источников — неприменимо.
- **MTProto собственной имплементации.** Переизобретение Telethon; риски выше, выгода отсутствует.
- **Парсинг веб-версии t.me.** Ограничено публичными каналами без истории; нет real-time.

## Последствия

- **Позитивные.** Полный доступ к целевым чатам; единый реальный account для чтения + enrichment (FEATURE-07 требует поиска общих чатов).
- **Негативные.** Риск бана аккаунта (ТЗ разд. 8). Митигация: ротация аккаунтов, мониторинг FloodWait, runbook `telethon-ban.md`. Серые зоны ToS Telegram — принимаем.
- **Операционные.** Session-файл — критичный секрет, шифруется на диске, бэкапится офлайн; потеря → пересоздание с новым номером.
