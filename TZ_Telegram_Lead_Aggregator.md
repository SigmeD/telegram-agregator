## 1. Бизнес-контекст

### 1.1. Цель продукта

Автоматизировать поиск потенциальных клиентов (основателей стартапов, которым нужна команда разработки MVP) через мониторинг открытых Telegram-чатов и каналов стартап-сообществ СНГ.

### 1.3. Ключевые пользователи

- **Основной пользователь:** Максим (PM, принимает решения по лидам)
- **Вторичный:** будущие sales-менеджеры команды

```
┌─────────────────────────────────────────────────────────────┐
│                 TELEGRAM LEAD AGGREGATOR                     │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  [Telethon Listener]  →  [Message Queue]  →  [Processor]    │
│         ↓                      ↓                   ↓         │
│   User-session          Redis/RabbitMQ      LLM Classifier  │
│   Читает чаты           Буфер сообщений     Claude/GPT-4    │
```

### 2.2. Технологический стек

**Backend:**
- Python 3.11+ (основной язык)

- Celery (фоновые задачи)

**AI/LLM:**
- Anthropic Claude API (основной) — для классификации и извлечения данных
- OpenAI GPT-4 (резервный)

**Frontend (админ-панель):**
- Next.js 15 + React
- Tailwind CSS + shadcn/ui
- TanStack Table для списков лидов

**Инфраструктура:**
- Docker + Docker Compose
- VPS (минимум 4 GB RAM, 2 CPU — Telethon требователен)
- Nginx как reverse proxy
- GitHub Actions для CI/CD

**Бот уведомлений:**
- Aiogram 3.x (отдельный бот для уведомлений Максиму)

### 2.3. Принципиальные архитектурные решения

1. **User-session, не Bot API.** Боты не могут читать чаты, в которые их не добавили. Нужен полноценный user-аккаунт Telegram с отдельным номером (лучше eSIM или виртуальный номер с долгой историей).
2. **Асинхронная обработка.** Telethon слушает, складывает в очередь, Celery воркер обрабатывает LLM-ом. Это защищает от rate-limit ошибок и пиковых нагрузок.
3. **Два уровня фильтрации.** Первый — по ключевым словам (дешёвый, быстрый). Второй — LLM-классификация (дорогой, точный). Это снижает расходы на API в 10–20 раз.
4. **Soft delete и аудит.** Все сообщения сохраняются с метаданными — можно переобучать фильтры задним числом.

### FEATURE-01: Подключение и авторизация Telegram-аккаунта

**Приоритет:** P0 (без этого ничего не работает)
**Оценка:** 2–3 дня

**Описание:**
Настройка user-session Telethon для чтения чатов от имени реального аккаунта.

**Требования:**
- Получение API_ID и API_HASH с my.telegram.org
- Регистрация отдельного Telegram-аккаунта (отдельный номер)
- Генерация session-файла (хранится в зашифрованном виде)
- Механизм обновления сессии без остановки сервиса
- Защита от FloodWaitError и автоматический retry с экспоненциальным backoff
- Логирование всех API-вызовов

- [ ] Аккаунт стабильно работает 7+ дней без ban/restriction
- [ ] Автоматический reconnect при разрыве соединения
- [ ] Graceful shutdown с сохранением состояния
- [ ] Логи сохраняются и ротируются (не более 100 MB на файл)

**Риски:**
- Telegram может блокировать аккаунты, которые читают слишком много. Решение: использовать 2–3 аккаунта в ротации, ставить задержки, не делать массовых действий.

### FEATURE-03: Telethon Listener (приём сообщений)

**Приоритет:** P0
**Оценка:** 3–4 дня

**Описание:**
Основной воркер, который слушает все подписанные чаты в реал-тайме и складывает новые сообщения в очередь.

**Требования:**

**Логика работы:**
1. При старте сервиса — подключение к Telegram через Telethon
2. Регистрация обработчика `@client.on(events.NewMessage)` на все активные источники
3. При получении сообщения — извлечение метаданных и отправка в Redis queue
4. Параллельная запись сырого сообщения в `raw_messages` таблицу

**Модель `raw_messages`:**
```sql
CREATE TABLE raw_messages (
    id UUID PRIMARY KEY,
    source_id UUID REFERENCES telegram_sources(id),
    telegram_message_id BIGINT NOT NULL,
    sender_id BIGINT,
    sender_username VARCHAR(100),
    sender_name VARCHAR(500),
    message_text TEXT,
    has_media BOOLEAN DEFAULT false,
    media_type VARCHAR(50),
    reply_to_message_id BIGINT,
    thread_id BIGINT,
    sent_at TIMESTAMP NOT NULL,
    received_at TIMESTAMP DEFAULT NOW(),
    processing_status VARCHAR(50) DEFAULT 'pending',
    -- 'pending', 'filtered_out', 'analyzing', 'lead', 'not_lead', 'error'
    UNIQUE(source_id, telegram_message_id)
);

CREATE INDEX idx_raw_messages_status ON raw_messages(processing_status);
CREATE INDEX idx_raw_messages_sent_at ON raw_messages(sent_at DESC);
```

**Требования к производительности:**
- Обработка до 1000 сообщений/минуту без потерь
- Latency от получения до записи в БД — < 500 мс
- Потребление RAM — не более 1 GB на воркер

**Обработка ошибок:**
- `FloodWaitError` → ждать указанное время + 10%
- `ChannelPrivateError` → пометить источник как неактивный, уведомить админа
- `AuthKeyError` → остановить сервис, уведомить критическим алертом
- Сетевые ошибки → retry с backoff до 5 попыток

**Критерии приёмки:**
- [ ] Uptime > 99% за 7 дней тестирования
- [ ] Нет потерь сообщений при рестарте сервиса
- [ ] Корректная работа с тредами и ответами
- [ ] Метрики Prometheus: messages_received_total, messages_per_minute

**Прямые запросы (вес 10):**

- "ищу разработчиков"

- "порекомендуйте команду"
- "looking for developers"
- "need CTO"

**Болевые сигналы (вес 7):**
- "наш подрядчик"

- "разочаровался"

**Логика:**
1. Сообщение приходит в фильтр

4. Если сумма весов >= 5 → передать в LLM для детального анализа
5. Иначе → `filtered_out`

**Критерии приёмки:**
- [ ] Точность фильтра (precision): > 70% сообщений, прошедших фильтр, реально попадают в LLM-оценку как потенциальные лиды
- [ ] Полнота (recall): не более 10% реальных лидов теряются на этапе keyword-фильтра
- [ ] Обработка 1 сообщения — < 50 мс
- [ ] Словарь триггеров редактируется через админ-UI

### FEATURE-05: LLM-классификация и извлечение данных

**Приоритет:** P0
**Оценка:** 4–5 дней

**Описание:**
Анализ каждого отфильтрованного сообщения через LLM для извлечения структурированной информации и финальной оценки релевантности.

**Требования:**

**Промпт (базовая версия):**
```
Ты — аналитик лидов для команды разработки MVP.

Твоя задача: проанализировать сообщение из Telegram-чата основателей стартапов
и определить, является ли автор потенциальным клиентом для команды, которая
разрабатывает MVP за 2-3 месяца с использованием AI-инструментов.

Контекст сообщения:
- Чат: {chat_title}
- Категория чата: {chat_category}
- Автор: {sender_name} (@{sender_username})
- Дата: {sent_at}
- Текст: {message_text}
- Ответ на сообщение: {reply_context}

Верни JSON со следующей структурой:
{
  "is_lead": boolean,
  "confidence": 0.0-1.0,
  "lead_type": "direct_request" | "pain_signal" | "lifecycle_event" | "not_a_lead",
  "stage": "idea" | "pre_mvp" | "mvp" | "growth" | "unknown",
  "urgency": "high" | "medium" | "low",
  "budget_signals": "mentioned" | "implied" | "none",
  "vertical": "fintech" | "saas" | "marketplace" | "edtech" | "other" | "unknown",
  "extracted_needs": "string — что конкретно ищет автор",
  "recommended_action": "contact_now" | "contact_soon" | "monitor" | "ignore",
  "recommended_approach": "string — как лучше зайти с учётом контекста",
  "red_flags": ["string"] — причины не работать с этим лидом,
  "reasoning": "string — краткое обоснование решения"
}

Правила:
1. is_lead=true только если автор реально ищет команду/разработчика
2. Если автор сам — разработчик/дизайнер/подрядчик → is_lead=false
3. Вопросы "а кто-нибудь делал X" без намерения заказать → is_lead=false
4. confidence < 0.6 → recommended_action=monitor
5. Прямой запрос на разработку MVP → confidence > 0.8
```

**Модель `lead_analysis`:**
```sql
CREATE TABLE lead_analysis (
    id UUID PRIMARY KEY,
    raw_message_id UUID REFERENCES raw_messages(id),
    is_lead BOOLEAN NOT NULL,
    confidence DECIMAL(3,2),
    lead_type VARCHAR(50),
    stage VARCHAR(50),
    urgency VARCHAR(20),
    budget_signals VARCHAR(20),
    vertical VARCHAR(50),
    extracted_needs TEXT,
    recommended_action VARCHAR(50),
    recommended_approach TEXT,
    red_flags JSONB,
    reasoning TEXT,
    llm_model VARCHAR(100),
    tokens_used INT,
    cost_usd DECIMAL(10, 6),
    analyzed_at TIMESTAMP DEFAULT NOW()
);
```

**Требования к производительности:**
- Среднее время анализа — < 3 сек
- Стоимость анализа — < $0.005 за сообщение (при Claude Haiku)
- Fallback на GPT-4 при недоступности Claude API
- Батч-обработка: до 5 сообщений параллельно

**Контроль качества:**
- Еженедельно Максим вручную проверяет 20 случайных решений LLM
- Если precision < 80% — корректировка промпта
- История промптов версионируется

**Критерии приёмки:**
- [ ] Precision классификатора > 80% на validation-наборе (100 размеченных вручную сообщений)
- [ ] Recall > 90%
- [ ] Средняя стоимость на 1 проанализированное сообщение < $0.005
- [ ] Все анализы сохраняются с reasoning для аудита

### FEATURE-06: Скоринг и ранжирование лидов

**Приоритет:** P1
**Оценка:** 2 дня

**Описание:**
Финальный скор от 0 до 100 для каждого лида, учитывающий множество факторов.

**Требования:**

**Формула скоринга:**

```python
def calculate_lead_score(lead_analysis, source, sender_history):
    score = 0

    # LLM confidence (до 40 баллов)
    score += lead_analysis.confidence * 40

    # Тип лида (до 20 баллов)
    type_scores = {
        "direct_request": 20,
        "pain_signal": 15,
        "lifecycle_event": 10,
        "not_a_lead": 0
    }
    score += type_scores[lead_analysis.lead_type]

    # Срочность (до 15 баллов)
    urgency_scores = {"high": 15, "medium": 8, "low": 3}
    score += urgency_scores[lead_analysis.urgency]

    # Приоритет источника (до 10 баллов)
    score += source.priority

    # Бюджетные сигналы (до 10 баллов)
    budget_scores = {"mentioned": 10, "implied": 5, "none": 0}
    score += budget_scores[lead_analysis.budget_signals]

    # История автора (до 5 баллов)
    if sender_history.is_founder_profile:
        score += 5
    if sender_history.previous_leads_count > 0:
        score -= 10  # уже обрабатывали

    return min(100, max(0, score))
```

**Категории по итоговому скору:**
- **Hot (80–100):** немедленное уведомление Максиму в Telegram + email
- **Warm (60–79):** попадает в дневной дайджест
- **Cold (40–59):** еженедельный отчёт
- **Irrelevant (<40):** в архив без уведомлений

**Критерии приёмки:**
- [ ] Скоринг работает за < 10 мс
- [ ] Корреляция скора с реальной конверсией в звонок > 0.5 (после 2 месяцев данных)
- [ ] Веса факторов настраиваются через конфиг (не хардкод)

Список Warm лидов за последние 24 часа с кратким описанием.

3. **Недельный отчёт (понедельник 10:00):**
Статистика: сколько сообщений обработано, сколько лидов найдено, конверсии.

**Inline-кнопки:**
- "Открыть в Telegram" → deep link на сообщение
- "Открыть в CRM" → ссылка на админ-панель
- "Пометить как обработан" → меняет статус
- "Не лид" → обучающий сигнал для системы (снижает вес похожих триггеров)

**Критерии приёмки:**
- [ ] Hot лид доходит до Максима за < 60 сек от момента получения сообщения
- [ ] Дайджест приходит ежедневно в заданное время
- [ ] Кнопки работают корректно
- [ ] Поддержка тихих часов (не беспокоить с 23:00 до 8:00, кроме критических лидов)

### FEATURE-09: Админ-панель

**Приоритет:** P1
**Оценка:** 5–7 дней

**Описание:**
Web-интерфейс для управления системой, просмотра лидов и аналитики.

**Требования:**

**Страницы:**

1. **Dashboard (главная):**
   - KPI-карточки: лидов сегодня/за неделю/за месяц
   - График: динамика лидов по дням
   - Топ-10 источников по конверсии
   - Последние 10 горячих лидов

2. **Лиды:**
   - Таблица с фильтрами (статус, score, источник, дата)
   - Карточка лида с полным контекстом
   - Поля статуса воронки: new → contacted → in_dialog → qualified → proposal → won/lost
   - Заметки и история взаимодействий

3. **Источники:**
   - CRUD для Telegram-чатов
   - Статистика по каждому источнику
   - Возможность массового импорта из списка

4. **Триггеры:**
   - CRUD ключевых слов
   - A/B тестирование новых триггеров
   - Аналитика эффективности

5. **Аналитика:**
   - Воронка лидов
   - Стоимость лида по источникам
   - Качество LLM (precision/recall на размеченных данных)
   - Cost tracking (расходы на API)

6. **Настройки:**
   - Управление аккаунтами Telegram
   - Промпты для LLM (с версионированием)
   - Веса скоринга
   - Каналы уведомлений

**Технические требования:**
- Авторизация: JWT + 2FA для админа
- Responsive дизайн (работает на планшете)
- Экспорт лидов в CSV/Excel
- API-интеграция с HubSpot/amoCRM (для будущего)

**Критерии приёмки:**
- [ ] Все CRUD-операции работают
- [ ] Страницы загружаются < 2 сек
- [ ] Таблица лидов поддерживает 10 000+ записей без замедления
- [ ] Мобильная версия функциональна

### FEATURE-10: Защита от детектирования и банов

**Приоритет:** P1
**Оценка:** 2–3 дня

**Описание:**
Комплекс мер, чтобы Telegram не счёл наш user-аккаунт ботом и не забанил.

**Требования:**

1. **Имитация человеческого поведения:**
   - Случайные задержки между действиями (1–5 сек)
   - Не более 200 операций API в минуту
   - Чтение сообщений с естественной периодичностью (не опрос каждые 100 мс)

2. **Профиль аккаунта:**
   - Реалистичная аватарка
   - Заполненное bio
   - Минимум 30 подписок на разные каналы (не только стартап-тематика)
   - Несколько контактов

3. **Ротация аккаунтов:**
   - 2–3 аккаунта, которые мониторят разные группы источников
   - Переключение при признаках rate-limit

4. **Мониторинг здоровья:**
   - Метрика "requests per minute" с алертом при 80%+ от лимита
   - Автопауза на 10 мин при первом warning от Telegram

5. **Контингенси-план:**
   - Процедура восстановления аккаунта при бане
   - Backup session-файлов
   - Документация по всем настройкам

**Критерии приёмки:**
- [ ] Аккаунт живёт без ban'а 30+ дней под нагрузкой
- [ ] Метрики здоровья выведены в дашборд
- [ ] Автоматическая ротация при достижении лимитов

## 9. Открытые вопросы для обсуждения

Эти пункты нужно решить до старта разработки:

1. **Какой API LLM использовать как основной?** Рекомендую Claude (Anthropic) из-за лучшего качества на русском и более дешёвого Haiku для классификации.

2. **Где хостить?** VPS (Hetzner/Timeweb/Selectel) или managed cloud (AWS/GCP)? Рекомендую Hetzner — в 3–5 раз дешевле при том же качестве.

3. **Нужна ли админ-панель сразу, или Telegram-бота достаточно на первый месяц?** Если бюджет сжатый — можно отложить FEATURE-09 и начать работать через бот.

4. **Как будем размечать данные для валидации LLM?** Нужен размеченный датасет из 100–200 примеров. Разметку делает Максим в первую неделю работы системы.

5. **Кто отвечает за поддержку после запуска?** Нужна процедура on-call на случай падений.
